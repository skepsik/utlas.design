## Что такое SemanticThread

`SemanticThread` — семантическая ветка разговора, собранная доменным сервисом из кандидатов-нод (`MessageRef`). Её задача — подсветить модели конкретный разговор на фоне общего шума чата.

В коде: тип `{ messages: MessageRef[] }` (`@utlas/core/domain/model/semantic-thread`).

`SemanticThread` ≠ reply-chain в Telegram. Reply-chain — одна из возможных эвристик, не определение понятия.

---

## Архитектура сборки

### Три слоя: Selector → Heuristic → Builder

Целевая модель (v0 — только первый слой, см. **Сейчас vs цель**):

```text
Trigger (qualifying-событие)
│
▼
Selector (storage/)     ← знает о способе хранения, не домен
│                       возвращает кандидатов (ноды)
▼
Heuristic (domain/)     ← детерминированные, возвращают claim/score
│                       оценивают каждую ноду
▼
Builder (domain/)       ← выбирает комбинацию selector+heuristic,
│                         возможны рекурсивные проходы
▼
(опционально) LLM-инференс ← финальная фильтрация кандидатов;
│                           claim/score от эвристик — подсказки
▼
SemanticThread          ← готовая собранная ветка
```

**Сейчас (v0):** `buildSemanticThread` в `@utlas/core/domain/services/build-semantic-thread` — монолитный builder: вызывает `readPort.selectors.replyChain.select({ anchor, transport })`, без отдельного Heuristic-слоя.

## Selector

Живёт в `storage/` — реализация напрямую связана со способом хранения, это не вопрос домена.

Контракт: `MessageSelector` — `{ id, select(ctx: SelectContext) }`; registry на порту — `MessageReadPort.selectors` (`MessageReadSelectors`).

| Селектор (цель) | Описание | Сейчас (v0) |
| ----------------- | -------- | ----------- |
| `recentN` | Последние N нод от триггера | `windowBefore(limit)` → `createWindowBeforeSelector`; доменный сервис `selectRecentBefore` для slot **CHAT HISTORY** |
| `anchorRefChain` | Цепочка по полю `MessageRef.anchorRef` от триггера | `replyChain` → `createReplyChainSelector`; slot **SEMANTIC THREAD** в `buildSemanticThread` |

TTL reply-chain: **нет** — история слепок, не устаревает. Selector в домене **не** живёт (только `storage/`).

Предполагаемые селекторы (из utlas-bot#42, частично устарело)

| Сценарий                              | Selector(s)                          | Ожидание                   |
| ------------------------------------- | ------------------------------------ | -------------------------- |
| Reply на старое со ссылкой вне окна N | `getReplyNodes`                      | ref со ссылкой в shortlist |
| Ссылка → @bot → ответ → reply на bot  | `getReplyToBotBranch`, link selector | ref со ссылкой включён     |
| @ + reply на user (fork)              | TBD selector/heuristic               | ветка с bot                |
| Burst без reply (private)             | `getOpenUtteranceNodes`              | utterance одного sender    |
| Group: дописка без @                  | `getOpenUtteranceNodes`              | same utterance (#37)       |
| Quote без reply                       | `getReplyNodes` / anchor             | excerpt + anchor           |

## Якорь

**Turn anchor** — `MessageRef` в `SelectContext.anchor` / `TurnRequest.anchor` (центр **USER MESSAGE** в prompt). Якорем может быть **любое сообщение** — не только user message.

**`anchorRef`** — поле `MessageRef`: transport-сигнал reply-parent; по нему ходит `replyChain` / целевой `anchorRefChain`.

Якорь определяет стартовую точку для selector `anchorRefChain`. Конкретный выбор якоря — ответственность контекста применения (Builder решает, какой selector/heuristic использовать).

## Эвристики

### Общий принцип: recall > precision

Лучше захватить лишнее, чем потерять нужное. Финальная фильтрация — задача LLM-шага (если он есть), а не эвристик. Векторный поиск / эмбеддинги **не** планируются.

### Возвращаемое значение: claim/score

Каждая эвристика возвращает `claim/score` — оценку релевантности ноды. Финальный score при нескольких эвристиках — см. **Открытые вопросы**.

### Типы эвристик

| Тип | Применяется к |
| --- | ------------- |
| Универсальная | Любая нода, смотрит на расположение |
| Текстовая | Текстовые сообщения |
| Документная | Сообщения с документами |

### Примеры эвристик

- Сообщение рядом по времени с триггером
- Общие обращения (к тому же пользователю / боту)
- Документ, сразу после которого началась цепочка ответов или обращение к боту
- Reply-chain по `anchorRef`

## Builder

- Выбор комбинации selector + heuristic — **исключительно ответственность Builder'а**
- Возможны рекурсивные проходы (точная алгоритмика — **Открытые вопросы**)
- Builder — доменный сервис, не знает о Telegram/transport

**Сейчас (v0):** роль builder выполняет `buildSemanticThread` (без выбора heuristic).

## LLM-фильтрация (опционально)

После всех прогонов возможен отдельный LLM-инференс для финальной фильтрации кандидатов. `claim/score` от эвристик передаются как подсказки. Builder отдаёт готовую ветку.

## Подача модели (concept)

**Не план реализации** — зафиксированные рассуждения о том, как `SemanticThread` попадает в turn prompt. Сборка ветки (Selector → Builder) здесь не меняется; спор только о **presentation layer** в compose.

Wire v0 — [turn-prompt](./turn-prompt.md) § User chain.

### Инварианты

- `SemanticThread` — **подмножество** тех же `MessageRef`, что попадают в окно контекста; хронология **та же**, что в ленте.
- Окно расширяется, если ветка **глубже** текущего `recentN` — prepend / expand до покрытия всех thread-нод (как сегодня prepend reply-parent в **CHAT HISTORY**).
- В один момент актуальна **одна** ветка для текущего **USER MESSAGE** — либо anchor ∈ thread, либо thread пуст.
- Несколько reply-веток «одновременно в фокусе» не бывает: fork / open utterance / deixis без `anchorRef` — **задача домена** (Builder), вычислить `Set<messageId>`; модель не выбирает между ветками.
- **`[reply to]`** — transport-маркер «на что ответили» (ребро, зафиксировал участник); **`[THREAD]`** (concept) — «эта реплика ∈ активной ветке» (членство, достроила фича). **Дополняют, не конкурируют:** thread докладывает связность там, где участник явно не связал; `[reply to]` не трогается. Для модели оба — аннотации **одного класса**; эпистемический статус (человек vs алгоритм) на силу сигнала в контексте не влияет.

Режим «только thread + **USER MESSAGE**» без полной ленты **не планируется**. Если понадобится — это **filtered feed**: то же окно, отфильтрованное по thread-маркерам, а не второй источник данных.

### Два режима presentation

| Режим | Суть | Сейчас (v0) |
| ----- | ---- | ----------- |
| **Annotated feed** | одна лента **CHAT HISTORY**; реплики ∈ thread помечены `[THREAD]`; policy: reasoning по маркеру | нет |
| **Isolated block** | contiguous extract в `=== SEMANTIC THREAD ===`; лента — фон | да (`semanticThreadResolver`) |

Policy для модели по смыслу один: **фокус reasoning на активной ветке текущего turn'а** — меняется только указатель (блок → маркер).

### За annotated feed

- Thread — **annotation**, не duplicate content: те же сообщения, та же хронология; второй блок с полным текстом не добавляет семантики, только токены.
- Лента сохраняет **контекст между узлами ветки** (side-talk, смена темы) — иногда нужен для ответа; isolated block это теряет.
- Модель собирает ветку фильтрацией по стабильному маркеру + `[reply to]` — не сложнее, чем читать отдельную секцию.
- **Аналогия:** `[reply to]` живёт маркером в ленте, отдельный блок для reply-связей никому не нужен — `[THREAD]` того же сорта; если маркерной аннотации хватает для `[reply to]`, нет причин требовать блок для thread.
- Out-of-window: корень ветки — первое `[THREAD]` в расширенном окне; **USER MESSAGE** — продолжение цепочки; выше watermark — не показано (как и при isolated block).

### За isolated block

Аргумент сильнее в **шумном group** (много участников, разреженная ветка на длинной ленте):

- Сообщения разных людей дают **кросс-помехи**: внимание разрывается между семантикой ветки и стилем/контекстом остальных.
- Thread-узлы в annotated feed **разнесены в token space**; связь между якорями может стать статистически слабее фона.
- **Соположение даёт готовую группировку** — флаг заставляет модель собрать рассеянные маркеры сквозь фон; contiguous extract может стоить дешевле по attention, чем сборка по `[THREAD]` под тяжёлым шумом (не «архивное ядро», а pre-assembled group).
- Duplicate block — не новая информация, а **salience / recitation**: вторая копия salient subset поверх той же ленты.

Аргументы **distance / lost-in-the-middle** — измерены на окнах **на порядок больше** utlas; на масштабе проекта (см. ниже) могут быть ниже порога заметности. **Framing-interference** (соседи перетягивают интерпретацию по содержанию, не по удалённости) блок **не убирает** — соседи остаются в ленте.

### Издержки isolated block

Если лента остаётся полной (иначе теряется чат), каждая реплика ∈ thread существует **в двух окружениях**:

- **раздвоение якоря** — модель должна сматчить два экземпляра как одну сущность;
- **двойной счёт** — реплика весит вдвое, искажая частотность; для фичи, цель которой восстановить контекст, это яд, а не feature.

Блок не снимает шум из context window — шум в history остаётся; выигрыш только если salience перевешивает издержки.

### Альтернативы блоку (concept)

Та же цель «сигнал над шумом» без дублирования:

- **взвешенный маркер** — не булев `[THREAD]`, а сильный признак (как `[reply to]` и цитата не «тонут» сами по себе);
- **прореживание** нерелевантного из окна (filtered feed) — сигнал без второго полнотекста.

### Масштаб окна (utlas)

Окно контекста — **десятки–сотня** сообщений (`contextLimitPrivate` / `contextLimitGroup` в compose), не тысячи. Lost-in-the-middle и distance-in-context болезненны на **абсолютных** дистанциях в десятки тысяч токенов; на utlas-размере та же **относительная** разреженность (`k / |window|`) может не давать заметной деградации.

Оси плотности и speaker entropy — **относительные**; деградация внимания — ещё функция **абсолютной** длины ленты в токенах. Рост окна или смена модели могут вернуть distance-эффекты; на текущем design-choice block может быть **преждевременной** оптимизацией.

### Реальные чаты vs synthetic worst case

Спор строился под синтетику: одна разреженная нить в **бесструктурном** фоне из девяти других тем. В живых лентах условия worst case **антикоррелированы**:

| Наблюдение | Следствие для block vs marker |
| ---------- | ------------------------------ |
| Далёкое → почти всегда через `anchorRef`, иначе не достучаться | «бессвязная далёкая нить» — артефакт синтетики |
| Горячая рабочая тема → лента сходится, между узлами ветки те же люди про то же | speaker interference как «чужие темы между узлами» слабее |
| Вялое многопоточное → **разные ветки**, не одна в шуме | параллельные структурированные нити, не homogeneous noise |

**Assumption:** block решает проблему, которую реальные чаты **структурно редко порождают**. Проверка — не GPU на синтетике, а **обсчёт логов**: доля далёких semantic-связей без `anchorRef`/цитаты. Если ≈ 0 — маркер достаточен by construction.

### Оси шума (не только N участников)

Число участников **N** — производная, не причина. Полезнее три оси:

| Ось | Смысл |
| --- | ----- |
| **Плотность** | `k / \|window\|` — доля ленты, которую занимает нить |
| **Дистанция разброса** | насколько далеко раскиданы звенья и сколько мусора между ними |
| **Структурная связность** | нить сшита `anchorRef` / цитатами vs держится только темой |

Пример: 10 участников + компактная плотная нить → маркер может хватить; 2 участника + огромные мусорные промежутки → соположение блока может помочь — но если звенья сшиты `[reply to]`, маркер **поверх рёбер** может сделать блок избыточным.

### Четыре зоны (рассуждение, не wire)

| Зона | Что работает | Block |
| ---- | ------------ | ----- |
| Структурно связная нить | `[reply to]` + `[THREAD]` | избыточен |
| Рёбра **потеряны** (transport обрезал) | чинить доставку | костыль под протечку |
| Бессвязно, но **тематически плотная** | маркер + контент; спорное — см. ниже | узкий «зазор» |
| Бессвязно, **разреженная** | retrieval («помнишь, мы говорили…») | не territory разметки |

Builder v0+ — **гибрид**: `anchorRef` как hard-сignal, семантическая плотность как soft-signal для достройки там, где цепь разорвана.

**«Зазор» блока** (плотно по теме, 0 рёбер): если связь достаточно **сильная**, чтобы builder сшил без `anchorRef`, она может быть достаточно **сильной**, чтобы модель увидела её из контента по флагу — тогда соположение не добавляет семантики. Если **слабая** для маркера — builder вряд ли сшил бы уверенно. Зазор может **логически схлопнуться**; единственное честное расхождение — эмпирика внимания, не диалектика.

### Ядро спора (concept)

Один бинарный вопрос, к которому диалог Utlas × Claude (2026-07) свёлся:

> **Пробивает ли флаг-помеченная семантическая близость шумовой фон без соположения, или под порогом шума ломается и требует физической смежности (блок)?**

Assumption по умолчанию для design: **annotated feed**; isolated block — гипотеза под synthetic worst case и большие окна, не установленная необходимость на utlas-масштабе. «Block единственный надёжный способ» — **Rejected**.

**Практика v0:** менять compose **не обязательно** — block не доказанно вреден, убрать — unproven risk; concept фиксирует направление, не mandate на diff.

### Протокол проверки (concept, optional)

Не план реализации; если когда-нибудь гонять — **одна критическая клетка**, не широкая сетка:

- нить **бессвязная** (0 `anchorRef`, 0 цитат — структура вырезана нарочно);
- **тематически плотная** (то, что hybrid-builder сшил бы семантикой);
- **растущий social noise**;
- три плеча, **равный token budget**: (A) duplicating block, (B) weighted marker + прореживание, (C) голая лента;
- ищем порог: ломается ли (B) раньше (A); (C) — помогает ли разметка вообще.

Остальные зоны (связная нить, потерянные рёбра, разреженный дрейф) — не block vs marker, а `reply to`, transport, retrieval.

### Ось выбора (concept, без метрик)

Жёсткого A/B в проекте не будет — проектировать по assumptions. Эвристика до проверки:

```text
threadDensity   = |thread| / |window|
scatterDistance = мусор между соседними thread-узлами
connectivity    = reply_to-сшитая | purely-thematic
```

| Сценарий | Склонность (assumption) |
| -------- | ------------------------ |
| structurally connected | annotated feed |
| private / duo, плотная нить | annotated feed |
| group, разреженная нить, высокий scatter | спорно; block не default без причины |
| purely-thematic, разреженная | retrieval, не presentation |

Presentation policy (TBD): `annotated_feed` | `isolated_block` — внутри **blob** wire; multi-message — [Rejected](./tools/native-tool-calls.md).

### Rejected (presentation)

- Два параллельных полнотекстовых представления **без причины** (и block, и маркеры на тех же репликах) — лишние токены, два источника правды, раздвоение якоря, двойной счёт.
- Отдельный блок как **единственный** способ подачи — постулат, не выдерживает учёт прореживания и взвешенного маркера.
- «Архивное ядро» как оправдание потери окружения — слабая метафора.
- Block как маска **потерянных рёбер** transport — лечить доставку, не дублировать.
- Широкий A/B «block vs marker вообще» — избыточен; если проверка, то одна клетка (бессвязно-плотное-под-шумом).
