# Utterance (концепт)

Зонтичный тип для всего что происходит в разговоре — текст, карта, документ. Домен работает с `Utterance`, не знает о деталях транспорта и хранения.

> **Статус:** концепт, не реализовано. Текущий код — `MessageRef` с опциональным `MessagePayload` (discriminated union). Переход к `Utterance` — отдельный design-review.

---

## Зачем

`MessageRef` начинает расширяться непрозрачно: `map_pin` payload сегодня, документы завтра. Discriminated union внутри одного типа неудобен в использовании — `switch` по `kind` везде, граница домена размытая.

Вопрос не в реализации, а в том **чьё это вообще**: карта — не детали транспорта, она существует в домене независимо от того умеет ли конкретный транспорт `sendLocation`. Где-то карта будет, где-то нет — в домене всегда одинаково.

---

## Идея

`Utterance` — корневой тип, конкретные виды — самостоятельные типы:

```ts
type Utterance =
  | { kind: "message"; ref: MessageRef }
  | { kind: "map_pin"; ref: MapRef }
  | { kind: "document"; ref: DocumentRef }
  // later: venue, poll, …
```

- `MessageRef` остаётся чистым — только текстовые сообщения
- Каждый вид самодостаточен, не мутирует общий тип
- Новый вид — новый тип, старый код не затрагивается
- `switch` по `kind` на верхнем уровне, не внутри `MessageRef`

Критерий попадания в `Utterance`: **меняет ли это семантику разговора для модели**. Карта была отправлена — модель должна это понимать как событие. Форматирование — нет.

---

## Хранение

Вопрос открытый. Документы потребуют FTS-индекса — отдельная таблица, своя схема, не jsonb.

Возможная структура:

```
utterances    — индекс: id, kind, conversation_id, created_at
messages      — текст (текущая таблица)
map_pins      — lat, lon, label, address
documents     — content, url, + FTS индекс
```

`utterances` как зонтик позволяет SemanticThread и CHAT HISTORY работать через единый индекс, не зная о конкретных видах. Конкретный вид читается только когда нужен.

`MessageRef` в домене возможно становится `UtteranceRef` — ссылка на любой вид, не только текст.

---

## Связь с текущим кодом

Сейчас в коде: `messages` — колонка **`type`** + **`payload`** jsonb; domain discriminant — `MessagePayload.type` (v0: `map_pin`). Работает для pin, но не для документов с FTS.

`utterances` как индексная таблица добавляется поверх существующей структуры — `messages` остаётся, `map_pins` и `documents` добавляются как отдельные таблицы по мере необходимости.

## Rejected

### `MessagePayload` как discriminated union внутри `MessageRef`

(**не развивать**): домен расширяется непрозрачно, `switch` по `kind` в каждом consumer, mutual exclusivity не очевидна. Работает для одного вида (`map_pin`), плохо масштабируется.

### Опциональные поля на `MessageRef`

`mapPin?: …`, `document?: …` (**не делаем**): union без контракта, компилятор не помогает, комбинации не ограничены.

### Single table inheritance

Одна таблица со всеми колонками, nullable для неприменимых видов (**не делаем**): антипаттерн, плохо масштабируется при добавлении видов.
